# HEARTBEAT.md

## HEARTBEAT.md и функция HEARTBEAT — подробное объяснение

### Что это такое

**HEARTBEAT** — это механизм периодической "пульсации" (polling), который запускает агента через заданные интервалы времени. Когда приходит heartbeat, агенту передаётся промпт из системных настроек (тот самый *"有 HEARTBEAT.md 就读（工作区上下文）..."*), и он может выполнить какие-то действия.

**HEARTBEAT.md** — это файл в рабочей директории агента, который определяет, *что именно* агент должен делать на каждом heartbeat.

### Как это работает

Пошагово:

1. Система (QwenPaw runtime) посылает heartbeat агенту с заданной периодичностью (настраивается в конфигурации агента).
2. Агент просыпается, читает **HEARTBEAT.MD**.
3. Если файл пустой или содержит только комментарии — heartbeat **пропускается** (агент ничего не делает, API-коллы не тратятся).
4. Если в файле есть задачи — агент выполняет их последовательно.
5. После выполнения агент может обновить HEARTBEAT.md (отметить выполненное, добавить новое).

### Зачем это нужно

#### 1. **Регулярные проверки без cron**

Вместо того чтобы создавать десятки cron-задач для разных проверок, можно объединить их в один heartbeat:

```
HEARTBEAT.md:
- Проверить почту (новые письма от клиентов)
- Проверить календарь (встречи на сегодня)
- Проверить статусы агентов (не завис ли кто)
- Проверить уведомления Telegram
```

Всё это выполняется **одним запуском агента**, а не несколькими отдельными cron'ами.

#### 2. **Контекстная осведомлённость**

При heartbeat агент видит актуальную директорию, файлы, память — он может принять решение на основе полного контекста, а не просто выполнить жёстко закодированную команду.

#### 3. **Экономия токенов/API**

- **HEARTBEAT.md пуст** → heartbeat ничего не делает, расходов нет.
- **HEARTBEAT.md с задачами** → один запуск вместо многих.
- **Cron** → каждый запуск отдельный, независимый.

### HEARTBEAT vs CRON — когда что использовать

| Критерий | HEARTBEAT | CRON |
|---|---|---|
| **Точность времени** | Плавающая (± несколько минут) | Точная (до минуты) |
| **Объединение задач** | ✅ Можно все проверки в одном heartbeat | ❌ Каждая задача — отдельный cron |
| **Разовые напоминания** | ❌ Не подходит | ✅ "Напомни через 20 минут" |
| **Контекст сессии** | ✅ Агент видит всю рабочую область | ⚠️ Ограниченный контекст |
| **Расход токенов** | Один запуск на все задачи | N запусков на N задач |
| **Примеры** | Проверить почту + календарь + статусы | "Каждый понедельник в 9:00 отчёт" |

**Оптимальная стратегия**: похожие периодические проверки объединять в HEARTBEAT, а точные или разовые события оставлять на cron.

### Примеры использования

**Пример 1: Пустой heartbeat (пропускается)**
```markdown
# HEARTBEAT.md — пусто, ничего не делаем
```

**Пример 2: Ежечасная проверка**
```markdown
# HEARTBEAT.md
- Проверить входящие письма от клиентов
- Проверить календарь на ближайшие события
```

**Пример 3: Условная логика**
```markdown
# HEARTBEAT.md
1. Прочитать MEMORY.md — если есть незавершённые задачи, продолжить
2. Проверить статусы агентов (chat_with_agent ql-hq)
3. Если рыночное время — проверить сигналы
```

### Как настроить

1. Убедись, что в конфигурации агента включены heartbeat'ы (настраивается в `config.json` или через API).
2. Отредактируй `HEARTBEAT.md` — добавь задачи, которые агент должен выполнять при каждом пульсе.
3. Если задач нет — просто оставь файл пустым или с комментариями: heartbeat будет пропускаться, экономя ресурсы.

### Коротко

> **HEARTBEAT.md** — это "список дел на каждый пульс".  
> Если список пуст — агент спит. Если есть задачи — просыпается и делает.  
> Экономит токены за счёт объединения проверок, но не подходит для точного расписания — для этого есть cron.
