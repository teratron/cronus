## Risks (Planning Audit)

- **Critical path = `crates/core` + `crates/nodus`**: Phases 3‚Äì8 depend on the library (Phases 1‚Äì2). Land the core contract and the nodus vertical slice early.
- **CLI-first surface (stem) is intentionally thin at Phase 3**: it ships the command framework + grammar + the commands available then; subsystem phases (4‚Äì6) attach their commands to it, and the TUI (Phase 7) mirrors the matured command set. This staging is the growth model, not scope creep.
- **nodus port size**: ~5k lines across six modules; Phase 2 builds it as a vertical slice (parse ‚Üí transpile ‚Üí minimal execute) before completing validator/lint and the full command set. Track parity against the reference test corpus.
- **Mobile/Tauri scaffold**: iOS/Android Tauri setup is toolchain-fragile (stack ¬ß5) ‚Äî smoke-test the build/sign pipeline in Phase 1, not at Phase 8.
- **Concept-only growth (v2.9.0 sync)**: 17 more specs absorbed (15 Stable L1 concept-only into Phase 0, 2 RFC into Backlog). - **Registry sync debt (resolved in v2.8.0)**: 48 specs had accumulated outside the plan (INDEX raced ahead to v1.0.36). Most have no authored L2 yet; they will surface as new phases via `/magic.spec` ‚Üí `/magic.task` when promoted.
- **Amendment deltas over Done phases (v2.10.0)**: the kanban 1.1.0 amendment (KAN-8 custom boards) is the first spec change landing after its implementation phase closed. Pattern: the delta becomes an explicit item in the next open phase (here Phase 10) rather than being silently absorbed or reopening the Done phase ‚Äî keeps Done phases immutable and the spec-vs-code gap visible.
- **TUI event-seam dependency (Phase 7)**: the render loop assumes a core event/subscribe seam. If the core exposes no pub/sub observer, the TUI must fall back to polling durable-state snapshots (INV-5 view-only). The Phase 7 tasks carry this fallback so the view panels are not hard-blocked on the subscription mechanism. Verify the seam first (T-7A02).
- **Core-decomposition wave sits in Backlog (v2.14.0)**: `l2-crate-topology` plus the `l2-source-layout` 1.2.0 delta are RFC, but they are not a feature ‚Äî they are a structural refactor of every crate in the workspace, and the topology spec states plainly that `l1-deployment-neutrality` DN-3 is *currently unrealizable* because the provider seams do not exist. (2) **Cost grows with delay**: every phase executed before the decomposition accretes more code inside the monolithic crate, so the refactor's size is a function of how long it waits at RFC.
- **First status regression on already-implemented code (v2.14.0)**: `l2-source-layout` went `Stable ‚Üí RFC` under the RULES ¬ß2 amendment rule *after* its Phase 1 implementation shipped. This extends the KAN-8 pattern ‚Äî that was an amendment to a spec over a Done phase; this one also **demoted the spec's status**, so the file no longer reads as `Stable` while the code it describes is live. The plan's response is unchanged (the Done phase keeps its `[x]` at delivered scope; the delta is tracked separately), but the failure mode is new: a later `/magic.task` could read the RFC status as "never implemented" and backlog the Phase 1 entry. The Backlog line states that boundary explicitly to prevent it.
- **Blocked archetypes are role-catalog debt, not an archetype-catalog defect (v2.14.0)**: `l2-archetype-catalog` ships one of the three archetypes it declares, because the preset role catalog is engineering-heavy and the other two name specialties it does not contain.
