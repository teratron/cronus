# Stage 19 Tasks — Loop Runner (L2)

**Phase:** 19
**Status:** Todo
**Strategic Goal:** Realize `l2-loop-runner` (LG-1…LG-10) — the engine-side mechanic that turns the loop-governance concept into a running component: a **loop runner** driving an autonomous loop turn after turn, plus a single **governor** enforcing the declared class, mutation manifest, oracle, and ceiling. It becomes the one place "run an agent in a loop" is implemented, so every loop inherits the same governance instead of each subsystem re-deriving its own retry-until-done logic. The load-bearing safety property is **LG-3 criteria immutability** — an actor never holds the right to edit the contract that judges it — enforced structurally by the type system (`MutableArtifact` has no `Criteria` variant), not by a runtime check that could be bypassed.

> **Real-integration, composition phase.** The runner does not re-implement isolation, budgeting, task selection, or model routing — it *composes* shipped subsystems (execution-workspace worktrees, budget-engine ceiling, model-router lineage, session-checkpoint state, version-control rollback, agent-autonomy approval gate, nodus provider traits) behind one governed loop contract. **Testability discipline (the acceptance spine, Phase 18 `SystemCalls`-seam precedent):** the pure loop *logic* — the governor's ceiling/guard/judge decisions, the execution runner's attempt→rollback→retry control flow, the escalation gate's held-out promotion — is CI-tested against **fakes** (a scripted oracle, a fake workspace allocator, a fake ledger sink, a scripted turn-runner) using the abstractions the spec already provides (`Oracle` enum, `ProviderType workspace_kind`, `BudgetHandle`), so `cargo test` needs no real worktree or model call; the *real* wiring (a live execution workspace, a real model-router judge, a real budget handle) is isolated to the `crates/core` facade (`loop_bootstrap.rs`, D02) plus one real end-to-end CLI smoke over a trivial deterministic-oracle unit. **Critical path = T-19A01** (the `LoopSpec`/`MutationManifest`/`Oracle` type shape — every track builds on it; keep it small, and LG-3-by-construction is the load-bearing invariant). **Hidden dependency:** the runner composes many existing subsystems — a composition-target API surprise is contained to the facade wiring (D02), never spread across the domain logic, because the domain tier tests against the seam abstractions. **Scope boundary (disclosed, not dropped):** `l2-loop-runner` §6 calls for *migrating* the existing loops (orchestration `/goal`, kanban execution, deep-research, heartbeat) to route through the runner — that is a separate, incremental, behavior-preserving refactor per §6, **not** part of this phase. Phase 19 builds the governed runner as a first-class component + CLI + validation; the per-subsystem migration is a deferred follow-up. Acceptance = each of LG-1…LG-10 covered by a test + criteria structurally unreachable through the manifest + the ceiling fires independently of the actor and oracle + a failed attempt rolls back and only the compact status carries forward.

## Atomic Checklist

- [ ] [T-19A01] Loop domain types (§4.1, LG-1/LG-2/LG-3): in `crates/domain/loop/spec.rs` — `LoopClass {Execution, Evolution}`, `MutableArtifact {Scratch, Plan, Knowledge, ValidationInput, Prompt, Tools}` (**no `Criteria` variant** — LG-3 enforced by the type system), `MutationManifest {class, tier, mutable}`, `Ceiling {max_iterations, budget_ref, deadline, patience}`, `ObjectiveSlot {objective, progress}` (LG-10), the `Oracle {Deterministic, Judge, Human}` enum, and `LoopSpec {class, manifest, oracle, ceiling, workspace_kind, objective_slot}`; plus the cross-crate result types the CLI/facade name (`LoopOutcome`, `LedgerEntry`, `Verdict`). **Verify:** `cargo test -p cronus-domain loop::spec` — a `MutationManifest` cannot express `criteria` (no such variant; a test asserting the mutable set is a subset of the six real kinds, `Criteria` uncompilable — LG-3 by construction); constructing a `LoopSpec` requires a declared class (LG-1); a Tier-0 same-prompt manifest has an empty `mutable` set; round-trip of a manifest through the run-record form (LG-2 recoverable).
- [ ] [T-19A02] Governor — single enforcement point (§4.4, LG-2/LG-3/LG-4/LG-6): in `crates/domain/loop/governor.rs` — `check_ceiling(Ceiling, n) -> Continue | Stop(reason)` (iterations/budget/deadline/patience, independent of actor and oracle — LG-6), `guard_writes(&[Mutation], &MutationManifest) -> Result<(), GuardError>` (each mutation's kind must be in `mutable`, else `IllegalMutation` — LG-2; criteria unreachable — LG-3), `judge(&Oracle, &TurnResult) -> Verdict` (dispatch deterministic/judge/human, stamp `reduced_confidence` on actor==oracle lineage — LG-4). **Verify:** `cargo test -p cronus-domain loop::governor` — `guard_writes` rejects a write to a kind outside `mutable` and returns `IllegalMutation` (a recorded event, not a silent drop); `check_ceiling` fires `Stop` on max-iter / exhausted budget / passed deadline / patience regardless of the actor's or oracle's state; `judge` stamps `reduced_confidence=true` exactly when actor.lineage == oracle.lineage.
- [ ] [T-19B01] Execution-loop runner (§4.2, LG-5/LG-8): in `crates/domain/loop/execution.rs` — `run_execution(spec, unit) -> LoopOutcome`: allocate workspace → `reconstruct_context` from plan/status (fresh context, LG-5) → `run_turn` → `governor.guard_writes` → `oracle.judge` → `record_iteration` (append-only ledger, LG-8) → on done `finalize_workspace`, else `rollback_to_snapshot` (VC-4) + `append_status`. The actor's "I'm done" is advisory; only `oracle.judge` sets `verdict.done`. **Verify:** `cargo test -p cronus-domain loop::execution` (against fakes) — a failed attempt rolls back and the next iteration starts from a clean snapshot carrying only the compact status note (LG-5); the ceiling stops the loop independent of the actor; every applied mutation appends to the ledger (LG-8); an out-of-manifest write is rejected before it reaches the (fake) disk; a passing deterministic oracle ends the loop `Done`.
- [ ] [T-19B02] Oracle wiring (§4.5, LG-4/LG-9): in `crates/domain/loop/oracle.rs` — `Deterministic` (output-contract validators: schema / callable exit-code / test command), `Judge` (model-router selects a binding whose lineage ≠ the actor's; if none available, run with the actor's lineage and set `reduced_confidence=true`), `Human` (routes through the agent-autonomy approval gate; in a background/cron context, falls back to deterministic or stops — **never** silent self-approval). LG-9: prefer a `Deterministic` oracle for the inner done-check when a validator exists. **Verify:** `cargo test -p cronus-domain loop::oracle` — a deterministic validator is preferred over judge/human when present (LG-9, logged not blocked); the judge path records `reduced_confidence` when only one lineage is available and full-confidence when lineages differ; the human oracle in a non-interactive context never self-approves (falls back to deterministic or stops).
- [ ] [T-19C01] Evolution-loop runner (§4.3, LG-1-composition/LG-8): in `crates/domain/loop/evolution.rs` — `run_evolution(spec, harness, task_set, held_out) -> Harness` wrapping EVALUATE→ANALYZE→IMPROVE, **nesting `run_execution`** to score each candidate on the task set; the evaluation pipeline (oracle) stays frozen (never in `spec.manifest.mutable`); a `predicted_flip` set is scored into a `keep | revert | partial` verdict (LG-8 with prediction, DH-8); `transfer_valid` on held-out gates the return (HE-6). **Verify:** `cargo test -p cronus-domain loop::evolution` (against fakes) — a candidate is kept only on a `keep` verdict and reverted otherwise; a manifest-bounded IMPROVE mutation never touches criteria (`guard_writes` rejects it if attempted); predicted-vs-actual scoring records keep/revert/partial into the ledger; the nested execution loop scores a candidate without mutating its own task (inner-loop class fixed).
- [ ] [T-19C02] Escalation gate — criteria change / self-evolution (§4.6, LG-7): in `crates/domain/loop/escalate.rs` — `escalate(target_loop, change) -> Promoted | Rejected`: require a **separated oracle** (target.oracle.lineage ≠ actor.lineage), an **external-novelty** source handle (new tasks/inputs/feedback, not the loop's own output), and a **held-out** evaluation; promote IFF `delta_metric >= margin && regression <= bound`; record the promotion **or** the rejection either way. This is the only path by which a *lower* loop's criteria become mutable — never a side effect inside the judged loop. **Verify:** `cargo test -p cronus-domain loop::escalate` — promotes only on a held-out metric gain within the regression bound (recorded); rejects and records the attempt on a held-out failure; **refuses** to escalate at all when the oracle shares the actor's lineage or no external-novelty source is present (LG-7 preconditions are hard gates, not advisory).
- [ ] [T-19D01] LG-10 objective persistence across in-session reduction (§3 compliance row, §4.4 two-realizations): the continuous-session realization — when a `LoopSpec` carries an `objective_slot`, the governor **re-projects** the standing objective + progress cursor into every turn from the durable session-checkpoint slot as a protected region (`l1-context-compression` CC-9, re-materialized if evicted), captures durable progress to the ledger **before** any lossy in-place reduction (CC-10), and treats the ledger — not the compacted transcript — as authoritative afterward (`l1-development-workflow` DW-5); the objective is phrased idempotent/resumable. An absent `objective_slot` degrades to LG-5 discrete-iteration behavior. **Verify:** `cargo test -p cronus-domain loop` (or a core integration) — a continuous-session loop whose supporting detail is compacted mid-run re-reads the re-projected objective and **resumes** (does not restart or redo completed work); the progress cursor read after a reduction comes from the durable ledger, not the compacted transcript; a spec with no `objective_slot` runs as a discrete-iteration loop (LG-5 path only).
- [ ] [T-19D02] Facade wiring + command surface (§4.7/§4.8, INV-3 parity): `crates/core/loop_bootstrap.rs` hands the runner **real** subsystems — an execution workspace (worktree), a live `BudgetHandle` (budget-engine), a model-router binding (judge lineage), a session-checkpoint store (state externalization) — the adapter-touching composition the domain tier cannot hold; and `cronus loop run <unit-id> [--tier N] [--max-iter N] | evolve <harness-id> [--target <score>] | log <run-id> | show <run-id>` (verb-first per `l2-cli` §4; the library method is the source of truth; TUI slash mirror). **Verify:** `cargo test -p cronus-cli` — `loop run|evolve|log|show --help` exit 0; a real end-to-end execution loop over a trivial deterministic-oracle work unit runs to `Done` and its ledger is inspectable via `cronus loop log <run-id>`; the CLI and the library bind the **same** core call (no divergent logic — INV-3 parity), proven by the smoke exercising the real facade constructor.
- [ ] [T-19T01] Validation sweep (LG-1…LG-10): `crates/core/tests/loop_invariants.rs` — one named test per invariant through the **real facade export chain**: LG-1 an undeclared class is not runnable; LG-2 an out-of-manifest write is rejected and recorded; LG-3 criteria are structurally unreachable (no variant); LG-4 oracle separation + `reduced_confidence` on lineage match; LG-5 state reconstructs-not-inherits across an iteration; LG-6 the ceiling stops independent of actor and oracle; LG-7 escalation promotes only on held-out gain with a separated oracle + novelty; LG-8 the mutation ledger is append-only; LG-9 the deterministic oracle is preferred when present; LG-10 the objective survives in-session reduction and resumes. **Verify:** `cargo test --workspace` green (exit 0) + `cargo clippy --all-targets -- -D warnings` clean + `cargo fmt --all -- --check` clean.

## Tracks

- **A — Types + governor (foundation):** T-19A01 → T-19A02. Gates every other track — all runners call the governor's single enforcement point, and every type flows from `spec.rs`. The load-bearing move is LG-3-by-construction (no `Criteria` variant).
- **B — Execution runner + oracle:** T-19B01 → T-19B02. The core execution loop and the three oracle kinds; serialized (the runner calls the oracle abstraction B02 defines, but B01's control flow can land against a scripted oracle first).
- **C — Evolution runner + escalation:** T-19C01 → T-19C02. The higher loop that changes artifacts and the gate by which criteria ever become mutable; depends on B01 (nests `run_execution`).
- **D — LG-10 persistence + facade + command surface:** T-19D01 (objective re-projection) · T-19D02 (real subsystem wiring + `cronus loop` CLI). D02 is the only real-integration task; it depends on A/B (a runner to wire) and is where a composition-target surprise is contained.
- **T — Validation:** T-19T01. Last; sweeps LG-1…LG-10 through the real facade.

Foundation-then-parallel: Track A gates all; B and C build the two runner families over the governor + oracle abstractions (tested against fakes); D01 adds the continuous-session objective guarantee and D02 wires the real subsystems + CLI (the sole real-integration surface, where any composition-API surprise is isolated); T closes the phase. Crate placement follows the post-decomposition topology (`l2-loop-runner` §4.8): I/O-free runner/governor/types in `crates/domain`, adapter-touching composition in the `crates/core` facade, reused provider traits in `crates/nodus` — no new `crates/contract` seam required (the runner *consumes* existing subsystem APIs and nodus traits; it defines no new port for an adapter to implement). The §6 migration of the existing orchestration/kanban/heartbeat loops to route through the runner is a deferred, behavior-preserving follow-up, not part of this phase.
