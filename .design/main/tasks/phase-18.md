# Stage 18 Tasks — Service Activation (L2)

**Phase:** 18
**Status:** Todo
**Strategic Goal:** Realize `l2-service-activation` (BA-1…BA-11) — the per-OS mechanism by which a host becomes the always-on INV-4 hub. One governing constraint drives every choice: **the background engine runs as the user, never as a superuser**, so every mode reaches the same per-user state root (this is what rejects the `LocalSystem` Windows Service and the root `LaunchDaemon`). The load-bearing security property is **BA-4**: background activation is a `l1-security` SEC-10 authority grant the agent has **no write path to** — enforced structurally (no activation tool, no sandbox filesystem write path to the registration locations, no elevation the agent can supply), not by trusting the agent to abstain. State is **observed, never remembered** (BA-8): every read queries the OS, and where OS *registered* differs from OS *effective*, the weaker wins.

> **Real-integration phase (Phase 17 precedent), seam-first.** The OS registration touches registries, plists, and service managers — genuine platform I/O behind an adapter crate (`l2-crate-topology` §4.4(a)). Per `l2-service-activation` §5, land the **seam + probe first** (`capabilities()` + `observe()` + a no-op adapter) so BA-8/BA-10 are testable before any registration code exists; add the **BA-4 sandbox deny entries before the first `enable()`** (the barrier must precede the capability it guards); then login-scoped (unelevated) before system-scoped (elevation ceremonies), Windows last (the S4U launch-timing gate). **Testability discipline:** the raw OS calls sit behind a thin `SystemCalls` seam so the adapter *logic* — remove-then-register ordering, weaker-wins `observe()` derivation, state mapping — is CI-tested against a scriptable fake on any host; the actual registry/plist/`systemctl` calls are a thin layer validated on the platform where runnable (Windows partially real-testable here; macOS/Linux logic via the fake + deferred real validation). Acceptance = each of BA-1…BA-11 covered by a test + a failed transition converges on `Inactive` (never doubly-registered) + `observe()` never returns a remembered value + the agent has no structural write path to the activation plane.

## Atomic Checklist

- [ ] [T-18A01] Activation seam + mock adapter (§4.1, BA-8/BA-10): `ActivationMode {Login, System}`, `ActivationState {Inactive, Active(mode), RequiresApproval(mode), Unknown{reason}}`, `ActivationCapabilities` (per-mode supported / `Unsupported{reason}`), and the `ActivationRegistry` trait (`capabilities` / `observe` / `enable` / `disable`) in `crates/contract` — no OS types cross the seam. Ship a scriptable no-op/mock adapter for tests. **Verify:** `cargo test -p cronus-contract activation` — the four states round-trip; a mock host with no supervisor reports `Unsupported` (BA-10); an unqueryable facility yields `Unknown`, never `Active` (BA-8); `RequiresApproval` is representable and distinct from `Active`.
- [ ] [T-18A02] Activation policy state machine (§4.4/§4.6, BA-3/BA-5/BA-6/BA-7): domain-tier transition engine over the seam — mutual exclusion (enable removes the *other* mode first), **remove-then-register-verify** (`remove → observe()==Inactive? → register → observe()==Active?`), consent-content bookkeeping (BA-5 discloses autonomy level + spend ceiling, per-mode never inherited), and failure convergence (a failed/refused step leaves `Inactive`, never doubly-registered). **Verify:** test — `enable(System)` from `Login` removes+verifies Login before registering; a forced register-failure leaves the host `Inactive` (BA-1 default); `RequiresApproval` is reported as failure, not folded into success; consent for Login does not authorize System.
- [ ] [T-18B01] BA-4 structural barriers (§4.5): add the registration-location **deny entries** (`HKCU\…\Run`, `~/Library/LaunchAgents`, `~/.config/systemd/user/`, `/etc/systemd/system/`, `~/.config/autostart/`, scheduled-task store) to the deny-by-default sandbox filesystem baseline (`l2-sandbox-policy` §4), and assert no activation tool is present in the allowlist-inverted tool surface (`l2-tool-security`). **Verify:** test — every registration location is absent from all `read_write` mounts of the sandbox scope, so agent-run code has no write path; no `activation`/`autostart` tool is exposed; the barrier precedes any `enable()` code path.
- [ ] [T-18C01] State-root lock — attach-never-duplicate (§4.7, BA-11): an exclusive lock on `state_root/engine.lock` taken at engine start, independent of activation mode, writing `{pid, endpoint, version}`; a second launch that cannot lock reads the record and **attaches** (version-compatible) or **fails visibly** (never starts a second engine); a stale lock is reclaimed through the existing liveness/crash-recovery path (WL-5 / CR-1), no new reclamation mechanism. **Verify:** test — two starts over one state root → the second attaches to the first's endpoint; an incompatible version → visible failure, no second engine; two engines over two distinct state roots coexist; a dead holder's lock is reclaimed.
- [ ] [T-18D01] Login-scoped OS adapters, unelevated (§4.2–§4.4, BA-2/BA-6/BA-8): the `activation-os` adapter's login path on all three platforms behind the `SystemCalls` seam — Windows `HKCU\…\Run` value + `StartupApproved` veto read; macOS `SMAppService.loginItem` + `status` mapping (`requiresApproval`→`RequiresApproval`); Linux systemd **user** unit, with the **XDG autostart** (`~/.config/autostart/*.desktop`) fallback on non-systemd hosts. Remove-then-register with read-back; no elevation on any platform. **Verify:** test — against the scriptable `SystemCalls` fake per platform, `observe()` derives the weaker-wins truth (veto/`requiresApproval`/`Hidden` beats the raw entry); `enable(Login)` writes only per-user locations and never elevates; real Windows login-scoped round-trips on the host where runnable.
- [ ] [T-18D02] System-scoped OS adapters, elevation-to-register-only (§4.2–§4.4, BA-6/BA-9): the system path — Windows **S4U scheduled task** (At-startup trigger, `RunLevel = Limited`; the launch-timing validation gate + `At log on` fallback per spec §5 note 7); macOS `LaunchDaemon` with `UserName = <user>` (`SMAppService.daemon`); Linux system unit with `User=<user>` **or** lingering user unit (both system-scoped, one mode). Elevation authorizes registration only; the engine runs unelevated; restart delegated to the supervisor (BA-9). Uninstall enumerates **all four** registration locations, removing only product-labelled ones (BA-7). **Verify:** test — a refused elevation leaves prior state intact (BA-6); registered process descriptor is unelevated; uninstall over the fake collects an orphan from a partially-failed transition; foreign (unlabelled) registrations are reported, never removed.
- [ ] [T-18E01] Command surface (§4.8, INV-3 parity): `cronus activation status|enable --mode <login|system>|disable` — verb-first per `l2-cli` §4; `status` prints `observe()`; interactive `enable` renders the BA-5 disclosure and requires confirmation; **non-interactive `enable` refuses** without an explicit acknowledgement flag (deliberate friction on the one command that grants unattended execution). TUI slash mirror + desktop settings pane bind the same core calls; the command is never an agent tool (§4.5). **Verify:** test — `activation status` prints the observed mode+state; `enable` in a non-interactive context without the ack flag exits non-zero and mutates nothing; the CLI/TUI/settings all bind the same core call (parity).
- [ ] [T-18T01] Validation sweep: BA-1…BA-11 invariant tests (observed-not-remembered, mutual exclusion, remove-then-register convergence on `Inactive`, per-mode spoke refusal, structural BA-4 no-write-path, attach-never-duplicate) + every OS adapter path has a `SystemCalls`-fake test for its remove-then-register ordering and weaker-wins observe. **Verify:** `cargo test --workspace` green + clippy `-D warnings` + `cargo fmt --all --check` clean.

## Tracks

- **A — Seam + policy (foundation):** T-18A01 → T-18A02. Gates B/C/D/E — every path calls the `ActivationRegistry` seam and the policy engine.
- **B — Structural barriers:** T-18B01. Must precede any real `enable()` (D) — the BA-4 barrier precedes the capability it guards.
- **C — State-root lock:** T-18C01. Independent of the rest, but a prerequisite for shipping *any* background mode (a supervised engine + a manual one over one state root is the exact BA-11 collision).
- **D — OS adapters (the real-integration core):** T-18D01 (login) → T-18D02 (system). Serialized on the shared `activation-os` crate + `SystemCalls` seam; Windows system-scoped last (S4U timing gate).
- **E — Command surface:** T-18E01. After A (policy) and D (a real path to drive).
- **T — Validation:** T-18T01. Last; sweeps every BA invariant.

Foundation-then-parallel: Track A gates everything; B and C proceed once the seam lands and both gate D; D is the platform-adapter core (serialized on the shared crate); E binds the surface; T closes the phase. Crate placement (the `activation-os` peer adapter crate, minted by `l2-crate-topology` §4.4(a)) is confirmed at execution and registered in `l2-crate-topology` via `/magic.spec` when it lands — the same pattern `model-local` followed, never a run-task spec edit.
